# Readiness-check persistence (Supabase + GitHub OAuth)

Server-side, append-only, tamper-proof storage of readiness-check attempts, with
instructor visibility. Students authenticate with **GitHub**; attempts are graded
and written **only** by the `grade-readiness` Edge Function, so a student can
never forge a score or delete their history.

## What's here

| Path | Purpose |
|---|---|
| `migrations/0001_readiness.sql` | Tables (`instructors`, `question_keys`, `readiness_attempts`), RLS, instructor rollup view |
| `functions/grade-readiness/index.ts` | The only writer: verifies JWT → grades server-side → inserts immutable attempt |
| `seed/question_keys.sql` | Answer keys (auto-generated by `scripts/build_readiness_keys.py`) |

## One-time provisioning (needs an Oxmiq owner)

1. **Create the Supabase project** (or self-host). Note the **Project URL** and
   **anon key** (Settings → API) — I need both to wire the front-end (task 5b).
2. **GitHub OAuth app** — GitHub → Settings → Developer settings → OAuth Apps →
   New. Authorization callback URL:
   `https://<project-ref>.supabase.co/auth/v1/callback`.
   Put the Client ID/Secret into Supabase → Authentication → Providers → GitHub.
3. **Apply the schema**: `supabase db push` (or paste `migrations/0001_readiness.sql`
   into the SQL editor).
4. **Seed answer keys**:
   ```bash
   python scripts/build_readiness_keys.py     # regenerates seed/question_keys.sql
   ```
   then run `seed/question_keys.sql` in the SQL editor. Re-run whenever a pool changes.
5. **Deploy the function**:
   ```bash
   supabase functions deploy grade-readiness
   ```
   It uses the `SUPABASE_URL`, `SUPABASE_ANON_KEY`, and `SUPABASE_SERVICE_ROLE_KEY`
   secrets that Supabase injects automatically.
6. **Add instructors** so they can read the cohort:
   ```sql
   insert into public.instructors (user_id, email)
   select id, email from auth.users where email = 'instructor@5ei.ai';
   ```

## Tamper-proofing summary (requirement 5)

- Clients have **no** INSERT/UPDATE/DELETE on `readiness_attempts` (no RLS policy
  grants it); only the service-role function writes → **append-only, immutable**.
- Grading is **server-side** against `question_keys` (RLS: zero client access) →
  scores **can't be forged**.
- Students SELECT only their own rows; instructors SELECT all → **private to the
  student, visible to instructors** for tracking growth across attempts.
